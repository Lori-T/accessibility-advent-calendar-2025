The elves have designed a stylish “gift wrapping machine control panel” with custom toggle switches. It looks good, but one of the toggles is not announced correctly by assistive technologies.

Your mission today

Check that custom components expose an accessible name, role and current value.

Quick 5-minute check

  • identify any non-standard controls (custom toggles, sliders, accordions, tabs)
  • check that they can be focused and operated with the keyboard
  • use a screen reader to see how they are announced
  • ensure they expose a role (for example, button, switch, tab) and a meaningful label

Test the wrapping machine panel

Below is a simplified panel. One of the controls is built using only a <div> without any role or label.

{# ❌ Deliberate issue: looks like a toggle but is just a div with no semantics #}
Festive ribbons on

Try using a screen reader or keyboard-only navigation. Only the real button is announced as an interactive control. The “Festive ribbons on” element is invisible to assistive tech.

The WCAG 2.2 skill behind this

4.1.2 name, role, value - user interface components must expose their role, name and current value to assistive technologies.

Why this matters

Custom controls that are not coded accessibly might look clickable, but are impossible to use for people relying on screen readers, switch devices or voice input.

Bonus challenge

Inspect the “Festive ribbons on” element in dev tools. How could you improve it (for example using a real <button> or adding role, aria-pressed and a label)?
